1. BIS Institutional Overview

The Bank for International Settlements (BIS) is an international organisation founded in 1930 and headquartered in Basel, Switzerland. It is owned by 63 central banks representing countries accounting for about 95% of world GDP. The BIS is often called "the central bank of central banks" because its primary customers are central banks and other official sector institutions.

What the BIS does

The BIS operates across three main areas:

  • Banking services — Takes deposits from ~200 central bank customers and invests the proceeds in high-quality assets. Total assets reached SDR 431.3 billion at end-March 2025. Annual Report p.102
  • Research & Statistics — Publishes global financial data (BIS statistics), coordinates regulatory standards (Basel Accords), and produces economic research via the BIS Working Papers.
  • Meeting services — Hosts meetings of central bank governors and international financial forums (FSB, BCBS, CPMI, IAIS).

Banking Department structure

The BIS Banking Department manages the institution's financial activities. Its activities include deposit-taking, gold and foreign exchange trading services, reserve management, and asset management. Annual Report p.101

Why this matters for the pipeline: Every SQL model in this project exists to analyse the Banking Department's currency deposit book — the borrowed funds placed by central banks with the BIS.

Unit of account — the SDR

The BIS reports all financial figures in Special Drawing Rights (SDR). The SDR is a basket of five currencies defined by the International Monetary Fund: USD, EUR, GBP, JPY, CNY. It serves as the Bank's unit of account, or "numeraire". Annual Report p.103

In the pipeline, all deal amounts are converted from their original currency into SDR using a monthly exchange rate table (src_fx_rates). This conversion happens in int_deals_with_sdr.

Currency deposits — the core product

Currency deposits are the main liability of the BIS Banking Department. They represent funds that central banks place with the BIS, similar to how retail customers deposit money at a commercial bank. The four deposit product types are: Annual Report p.105

FIXBIS_DEPFIXBISFixed-rate BIS deposit. Fixed term, fixed interest rate. Most common product.

BIS fiscal year

The BIS financial year runs from 1 April to 31 March. Annual Report p.101

  • FY2023-24 = 1 April 2023 → 31 March 2024
  • FY2024-25 = 1 April 2024 → 31 March 2025

This is why dim_date.fiscal_year uses April as the start of FQ1, not January.

Source system — Openlink Findur

The BIS booking system for Banking Department deals is Openlink Findur, a specialised treasury and trading platform used by banks and central banks worldwide. In this project, the CSV files (src_deals.csv, src_fx_rates.csv) simulate a flat file extract from Findur — the kind of file that would come out of a Findur report or scheduled export. Deal IDs follow the format FIN-YYYYMM-NNNNNN (FIN = Findur).

5. Star Schema Design

The data model follows a star schema: one central fact table surrounded by dimension tables, each joined via a foreign key. This is the standard pattern for analytical data warehouses and Power BI semantic models.

Why not a fully normalised (3NF) database?

Normalised databases (like a banking transaction system) avoid data duplication by splitting data into many tables. Great for transactional systems. Bad for analytics because you need many complex joins to answer a simple question. Star schemas denormalise intentionally — one join per dimension, predictable query patterns, fast aggregation.

Why not one wide table?

You could put everything into one table (the "one big table" anti-pattern). It would work but creates problems: repeated strings everywhere (counterparty name repeated 700 times per deal), no reusable calendar logic, no single place to update a dimension attribute (e.g., a counterparty's region), and Power BI cannot optimise relationships between facts and dimensions.

6. SQL Function Reference

Every T-SQL function used in this project, with a plain English explanation and a concrete example.

FunctionWhat it doesUsed inExample
TRY_CAST(x AS type) Converts x to the target data type. If the conversion fails (e.g. text that is not a valid date), returns NULL instead of throwing an error. Safer than CAST(). stg_deals, stg_fx_rates TRY_CAST('2023-04-03' AS DATE) → 2023-04-03
EOMONTH(date) Returns the last calendar day of the month containing date. No arguments needed for offset = 0. int_deals_with_sdr, mart_maturity_ladder EOMONTH('2023-04-15') → 2023-04-30
EOMONTH(date, n) Same but offsets by n months first. n = -1 means "the month before". int_deals_with_sdr EOMONTH('2023-04-15', -1) → 2023-03-31
DATEADD(part, n, date) Adds n units of part (day/month/year) to date. dim_date, mart_maturity_ladder DATEADD(month, 3, '2023-04-01') → 2023-07-01
DATEDIFF(part, start, end) Returns the difference between end and start in units of part. Note: start comes first. Positive if end > start. mart_maturity_ladder DATEDIFF(day, '2025-03-31', '2025-04-07') → 7
DATEFROMPARTS(y, m, d) Constructs a DATE from integer year, month, day components. Used to decode integer date keys back into real dates. mart_maturity_ladder DATEFROMPARTS(2025, 3, 31) → 2025-03-31
YEAR(date) / MONTH(date) / DAY(date) Extract the year, month, or day as an integer from a date. fct_currency_deposits, dim_date YEAR('2023-04-03') → 2023
DATENAME(part, date) Returns the name of the specified part as a string. dim_date DATENAME(month, '2023-04-03') → 'April'
DATEPART(part, date) Returns an integer for the specified part. DATEPART(quarter, …) returns 1–4. dim_date DATEPART(quarter, '2023-07-01') → 3
CONCAT(a, b, …) Joins strings together. NULLs are treated as empty strings. dim_date CONCAT('FY', 2023, '-', 24) → 'FY2023-24'
RIGHT(str, n) Returns the rightmost n characters of a string. dim_date RIGHT(2024, 2) → '24'
ROW_NUMBER() OVER (ORDER BY …) Window function — assigns a sequential integer to each row. Used here to generate a sequence of numbers for date spine generation. dim_date Produces 0, 1, 2, … 729 for the 730-day date spine
CROSS JOIN Cartesian product — every row of table A is matched with every row of table B. If A has 700 rows and B has 24 rows, the result has 16,800 rows. mart_maturity_ladder 700 active deals × 24 month-end dates = 16,800 (deal, as_of_date) pairs
Date key arithmetic:

LAYER 1 stg_deals

staging view   File: models/staging/stg_deals.sql

Purpose: Take the raw src_deals table and make it type-safe. Every column that arrived as a VARCHAR string is cast to its correct data type. No rows are removed, no aggregation. Staging models are purely about type safety and naming consistency.

Key operation — TRY_CAST: Instead of CAST(), we use TRY_CAST() everywhere. If a date string is malformed (e.g. 'not-a-date'), TRY_CAST returns NULL silently. CAST would throw an error and halt the entire pipeline.

Key derivation — rate_decimal: We compute rate_pct / 100 here (e.g. 2.50 → 0.025000). This is done once in staging so that every downstream model can use either form without recomputing.

stg_deals.sql — staging · view · source: src_deals
with source as (

    select * from {{ source('bis_raw', 'src_deals') }}

),

staged as (

    select

        -- identifiers
        deal_id,

        -- product / instrument
        deal_type,

        -- counterparty
        cpty_code,
        cpty_name,
        cpty_country,

        -- dates: cast from string YYYY-MM-DD to DATE
        try_cast(trade_date    as date) as trade_date,
        try_cast(value_date    as date) as value_date,
        try_cast(maturity_date as date) as maturity_date,

        -- currency and amounts
        currency,
        try_cast(nominal_amount as decimal(20, 2))    as nominal_amount,

        -- interest rate: keep percentage form AND derive decimal form
        try_cast(rate_pct as decimal(10, 6))          as rate_pct,
        try_cast(rate_pct as decimal(10, 6)) / 100.0  as rate_decimal,

        -- deal lifecycle
        deal_status,

        -- internal classification
        portfolio,
        book,

        -- audit timestamps: cast from string to DATETIME2
        try_cast(created_dt       as datetime2) as created_dt,
        try_cast(last_modified_dt as datetime2) as last_modified_dt

    from source

)

select * from staged

LAYER 3 int_deals_with_sdr

intermediate view   File: models/intermediate/int_deals_with_sdr.sql

Purpose: Join the staged deals table to the staged FX rates table to compute nominal_amount_sdr — the deal value expressed in SDR, the BIS unit of account.

The join logic — explained step by step

The join has two conditions:

  1. Currency match — deals.currency = fx_rates.currency_code. We need the rate for the same currency as the deal.
  2. Date match — The rate used is the one for the last day of the month before the deal's value date. Formula: EOMONTH(DATEADD(month, -1, value_date)). Example: value date = 2023-04-15 → use rate for 2023-03-31.

Why the month before? This simulates a common practice in treasury systems: the FX rate used for SDR conversion is the rate that was known at the start of the month the deal settled, not the rate on the settlement date itself. This avoids look-ahead bias and matches how BIS calculates SDR amounts.

Why LEFT JOIN? If a deal's currency has no matching FX rate (e.g. a currency not in the rate table, or a rate date gap), the deal still appears in the output — but with nominal_amount_sdr = NULL. This is intentional: we want to see missing rates as NULLs for investigation, not silently drop deals.

int_deals_with_sdr.sql — intermediate · view
with deals as (

    select * from {{ ref('stg_deals') }}

),

fx_rates as (

    select * from {{ ref('stg_fx_rates') }}

),

joined as (

    select

        d.deal_id,
        d.deal_type,
        d.cpty_code,
        d.cpty_name,
        d.cpty_country,
        d.trade_date,
        d.value_date,
        d.maturity_date,
        d.currency,
        d.nominal_amount,
        d.rate_pct,
        d.rate_decimal,
        d.deal_status,
        d.portfolio,
        d.book,

        -- FX rate applied (kept for auditability)
        fx.rate_date                               as fx_rate_date,
        fx.sdr_per_unit,

        -- SDR conversion: multiply deal amount by the rate
        d.nominal_amount * fx.sdr_per_unit         as nominal_amount_sdr,

        d.created_dt,
        d.last_modified_dt

    from deals d

    left join fx_rates fx
        on  fx.currency_code = d.currency
        and fx.rate_date = eomonth(dateadd(month, -1, d.value_date))

)

select * from joined

LAYER 4 dim_counterparty

mart / dim view   File: models/mart/dim_counterparty.sql

Purpose: Build the counterparty dimension. Extract every unique counterparty from the deal book and add a region column derived from the ISO country code, using BIS's own geographic classification.

The region classification logic

A large CASE WHEN cpty_country = 'CN' THEN 'Asia-Pacific' … statement maps each ISO 3166-1 alpha-2 country code to one of six BIS regions. This matches Graph 4.3 in the BIS Annual Report. Annual Report p.106

The "Other" trap: The initial version only coded ~40 countries. Countries like Turkmenistan (TM), Macau (MO), Tajikistan (TJ) and 24 more were falling into the ELSE 'Other' bucket. Diagnostic query identified all 27 missing codes. They were added in the correct regional blocks. Result: zero rows in "Other" in Power BI.

dim_counterparty.sql — mart · view (abridged — showing structure)
with counterparties as (

    select distinct
        cpty_code,
        cpty_name,
        cpty_country
    from {{ ref('int_deals_with_sdr') }}

),

with_region as (

    select
        cpty_code,
        cpty_name,
        cpty_country,

        case cpty_country
            -- Asia-Pacific (61% of deposits)
            when 'CN' then 'Asia-Pacific'
            when 'JP' then 'Asia-Pacific'
            when 'IN' then 'Asia-Pacific'
            when 'KR' then 'Asia-Pacific'
            when 'AU' then 'Asia-Pacific'
            -- ... 33 more Asia-Pacific codes (TM, MO, TJ, BT, KZ, KG, CK, MV,
            --     WS, UZ, TL, SB, NP, VU, VN, LA, BN, TO, SG, HK, TH, MY,
            --     ID, PH, BD, PK, LK, MM, KH, MN, PG, FJ, NZ) ...

            -- Americas (9%)
            when 'US' then 'Americas'
            when 'CA' then 'Americas'
            when 'BR' then 'Americas'
            when 'CR' then 'Americas'  -- added: Costa Rica
            when 'HN' then 'Americas'  -- added: Honduras
            -- ...

            -- Europe (8%)
            when 'DE' then 'Europe'
            when 'FR' then 'Europe'
            when 'AZ' then 'Europe'    -- added: Azerbaijan
            when 'AM' then 'Europe'    -- added: Armenia
            when 'GE' then 'Europe'    -- added: Georgia

LAYER 7 dim_date

mart / dim view   File: models/mart/dim_date.sql

Purpose: Generate a calendar dimension covering every day in the data period (1 Apr 2023 → 31 Mar 2025 = 730 days). Each row has calendar attributes and BIS fiscal year/quarter attributes.

Date spine generation trick

On Fabric SQL Analytics Endpoint, you cannot use system tables like sys.all_objects to generate row sequences. Instead, we build a sequence by cross-joining CTEs to multiply row counts exponentially: 2 → 4 → 16 → 256 → 65,536. We take the top 730 rows and use ROW_NUMBER() to assign 0–729 as offsets, then add each offset to the start date with DATEADD(day, n, '2023-04-01').

BIS fiscal year formula

The fiscal year label is FY2023-24, FY2024-25, etc. The rule:

  • If MONTH(date) >= 4 → fiscal year starts in YEAR(date), e.g. April 2023 → FY2023-24
  • If MONTH(date) < 4 → fiscal year started the previous year, e.g. February 2024 → FY2023-24
dim_date.sql — mart · view · 730-day date spine
-- Cross-join trick to generate 65,536 rows without system tables
with
n1 as (select 1 as n union all select 1),
n2 as (select 1 as n from n1, n1 as b),       -- 4 rows
n3 as (select 1 as n from n2, n2 as b),       -- 16 rows
n4 as (select 1 as n from n3, n3 as b),       -- 256 rows
n5 as (select 1 as n from n4, n4 as b),       -- 65,536 rows

nums as (
    select top 730
        row_number() over (order by (select null)) - 1 as n
    from n5
),

date_sequence as (
    select dateadd(day, n, cast('2023-04-01' as date)) as full_date
    from nums
),

with_attributes as (

    select

        -- Primary key: pure arithmetic — avoids FORMAT() compatibility issue
        year(full_date) * 10000
        + month(full_date) * 100
        + day(full_date)                             as date_key,

        full_date,
        year(full_date)                              as year,
        month(full_date)                             as month_num,
        datename(month, full_date)                   as month_name,
        datepart(quarter, full_date)                 as quarter,

        -- BIS fiscal year: April to March
        case
            when month(full_date) >= 4
            then concat('FY', year(full_date), '-', right(year(full_date) + 1, 2))
            else concat('FY', year(full_date) - 1, '-', right(year(full_date), 2))
        end                                          as fiscal_year,

        -- BIS fiscal quarter: FQ1 = Apr/May/Jun, FQ2 = Jul/Aug/Sep, etc.
        case month(full_date)
            when 4  then 'FQ1' when 5  then 'FQ1' when 6  then 'FQ1'
            when 7  then 'FQ2' when 8  then 'FQ2' when 9  then 'FQ2'
            when 10 then 'FQ3' when 11 then 'FQ3' when 12 then 'FQ3'
            when 1  then 'FQ4' when 2  then 'FQ4' when 3  then 'FQ4'
        end                                          as fiscal_quarter

    from date_sequence

)

select * from with_attributes

LAYER 8 fct_currency_deposits

mart / fact view   File: models/mart/fct_currency_deposits.sql

Purpose: The central fact table of the star schema. One row per deal. Selects from int_deals_with_sdr and adds date keys (integer YYYYMMDD) for joining to dim_date.

Date key encoding

The formula YEAR(date) * 10000 + MONTH(date) * 100 + DAY(date) produces a sortable integer: e.g. 2 April 2023 → 20230402. This replaces FORMAT(date, 'yyyyMMdd') which is not available on Fabric SQL Analytics Endpoint.

Degenerate dimensions

Some attributes — deal_status, portfolio, book, fx_rate_date, sdr_per_unit — are kept directly on the fact table rather than having their own dimension table. These are called degenerate dimensions. The rule: if a column has too many values for a useful grouping, or if building a separate table would add complexity without analytical value, keep it on the fact.

LAYER 9 mart_maturity_ladder

mart / analytics view   File: models/mart/mart_maturity_ladder.sql

Purpose: A standalone analytics mart — not part of the star schema. It answers the question: "For any given month-end date, how much of the active deposit book matures within each time bucket?" This is the maturity ladder — a standard liquidity risk management report.

The problem this mart solves

A simple sum of deposits by product or region doesn't tell you when the money leaves. The maturity ladder shows the time distribution of future cash outflows: how much is due tomorrow (Overnight), in a week, in a month, etc. Risk managers and treasury teams use this daily.

Why CROSS JOIN?

We want to see the maturity profile at 24 different points in time — every month-end from April 2023 to March 2025. Instead of writing 24 separate queries, we cross-join all active deals with all 24 as-of dates. Each deal × each date produces one row. Then we calculate days_to_maturity from that specific date to the deal's maturity date. Result: 71 rows (24 dates × up to 7 buckets). PASS 1/1

The hybrid approach

The maturity bucket logic (which time window?) lives in dbt SQL. The choice of which date to view lives in Power BI as an as_of_date slicer. This is the right separation of concerns: SQL does the aggregation, Power BI does the interactivity.

Why is SIGHT_DEP always Overnight?

Sight accounts have no maturity date (maturity_date is NULL, so days_to_maturity would be NULL). They are redeemable at any time — the functional equivalent of "overnight". Rather than letting them fall through the bucket logic, we explicitly assign them to Overnight with WHEN deal_type = 'SIGHT_DEP' THEN '1. Overnight' as the first CASE branch.

mart_maturity_ladder.sql — mart · analytics view · 24 as-of dates × 7 buckets
with

-- Step 1: generate 24 integers (0 to 23)
month_nums as (
    select 0  as n union all select 1  union all select 2  union all select 3
    union all select 4  union all select 5  union all select 6  union all select 7
    union all select 8  union all select 9  union all select 10 union all select 11
    union all select 12 union all select 13 union all select 14 union all select 15
    union all select 16 union all select 17 union all select 18 union all select 19
    union all select 20 union all select 21 union all select 22 union all select 23
),

-- Step 2: convert each integer into a month-end date
-- EOMONTH('2023-04-01' + n months) = Apr 2023, May 2023, ... Mar 2025
as_of_dates as (
    select eomonth(dateadd(month, n, cast('2023-04-01' as date))) as as_of_date
    from month_nums
),

-- Step 3: filter to VALIDATED deals only, reconstruct maturity_date from key
active_deals as (
    select
        f.deal_id,
        f.deal_type,
        f.nominal_amount_sdr,
        f.maturity_date_key,

        -- Decode YYYYMMDD integer back into a DATE
        case
            when f.maturity_date_key is null then null
            else datefromparts(
                f.maturity_date_key / 10000,           -- year
                (f.maturity_date_key / 100) % 100,     -- month
                f.maturity_date_key % 100              -- day
            )
        end as maturity_date

    from {{ ref('fct_currency_deposits') }} f
    where f.deal_status = 'VALIDATED'
),

-- Step 4: CROSS JOIN — every active deal × every as-of date
-- Compute days_to_maturity from the as-of date to the deal's maturity
crossed as (
    select
        d.deal_id,
        d.deal_type,
        d.nominal_amount_sdr,
        d.maturity_date,
        a.as_of_date,
        datediff(day, a.as_of_date, d.maturity_date) as days_to_maturity
    from active_deals d
    cross join as_of_dates a
),

-- Step 5: assign each (deal, as_of_date) to a maturity bucket
with_buckets as (
    select
        as_of_date,
        case
            when deal_type = 'SIGHT_DEP'   then '1. Overnight'
            when days_to_maturity <=   7   then '2. <= 1 Week'
            when days_to_maturity <=  30   then '3. <= 1 Month'
            when days_to_maturity <=  90   then '4. <= 3 Months'
            when days_to_maturity <= 180   then '5. <= 6 Months'
            when days_to_maturity <= 365   then '6. <= 1 Year'
            else                                '7. > 1 Year'
        end as maturity_bucket,
        nominal_amount_sdr
    from crossed
    where days_to_maturity > 0    -- exclude matured deals
       or deal_type = 'SIGHT_DEP' -- sight accounts always included
),

-- Step 6: aggregate
final as (
    select
        as_of_date,
        maturity_bucket,
        count(*)                as deal_count,
        sum(nominal_amount_sdr) as total_sdr
    from with_buckets
    group by as_of_date, maturity_bucket
)

select * from final

Why SDR and not USD?

The BIS serves central banks globally. Converting to a single national currency (e.g. USD) would introduce FX volatility into the reporting — a drop in deposit volumes could just be the USD strengthening, not actual outflows. The SDR basket dampens this volatility and is the official BIS unit of account.

Why star schema and not flat tables?

Power BI's VertiPaq engine is optimised for star schemas. Relationships between one fact table and multiple dimension tables allow the engine to apply filters efficiently — a region filter on dim_counterparty propagates to fct_currency_deposits through the relationship, rather than requiring a SQL WHERE clause. This enables sub-second response on millions of rows.

Why views and not tables?

On Fabric SQL Analytics Endpoint, we cannot create physical tables (DDL restriction). Views are also more flexible during development — you can change the SQL and re-run dbt run without worrying about dropping and recreating physical tables. The trade-off: no pre-computed data, so every query re-reads from the underlying Delta files. For a ~10,000 deal book this is negligible; for tens of millions of rows, you would switch to Fabric Warehouse with physical tables.

10. Lexique / Glossary

SDR
Special Drawing Right. IMF basket currency (USD, EUR, GBP, JPY, CNY). BIS unit of account. Not a real currency — used for accounting and reporting.
BIS
Bank for International Settlements. Founded 1930, Basel. "Central bank of central banks." ~200 central bank customers. Assets: SDR 431.3bn (Mar 2025).
dbt (data build tool)
Open-source SQL transformation framework. Write SELECT queries; dbt handles CREATE/DROP, dependency resolution (DAG), and documentation. Jinja-SQL templating language.
DAG
Directed Acyclic Graph. In dbt, the dependency graph of all models. dbt resolves the correct execution order from ref() calls. No circular dependencies allowed.
CTE
Common Table Expression. A named temporary result set defined with the WITH keyword. Makes SQL readable by breaking a long query into named steps. Not persisted.
Star schema
Data modelling pattern: one central fact table + multiple dimension tables joined via foreign keys. Optimised for analytical queries and Power BI VertiPaq engine.
Fact table
The centre of a star schema. Contains measurable, additive business events (deals, transactions). Each row = one event. Foreign keys point to dimension tables.
Dimension table
Descriptive context for the fact table. Contains attributes used for filtering, grouping, and labelling. E.g. counterparty name, region, currency name.
Degenerate dimension
A dimension attribute kept on the fact table rather than in a separate dim table. Used when the attribute has too many values for a useful grouping (e.g. deal_id, portfolio, book).
Maturity ladder
A liquidity risk report showing how much of a portfolio matures in each time bucket (Overnight, ≤1W, ≤1M, etc.) at a given point in time. Used by treasury/risk teams to manage cash outflows.
FIXBIS
Fixed-rate Investment at the BIS. Fixed-term, fixed-rate currency deposit. Most common product in the BIS deposit book.
MTI
Medium-Term Instrument. Longer tenor (typically 1–10 years) fixed-rate deposit. Provides stable, long-term funding for the BIS.
Sight account
On-demand / overnight deposit. No fixed maturity. Central bank can withdraw at any time. Classified as Overnight in the maturity ladder.
Notice account
Deposit redeemable after a fixed advance notice period (e.g. 7, 30, 90 days). Between Sight and FIXBIS in terms of liquidity.
MS Fabric
Microsoft Fabric. Unified analytics platform on Azure. Components: OneLake (storage), Lakehouse (Delta/Parquet tables), SQL Analytics Endpoint (T-SQL queries), Power BI (reports).
OneLake
Microsoft Fabric's unified storage layer. All data (Delta/Parquet files) lives in OneLake. Multiple compute engines (SQL, Spark, Power BI) read the same files.
SQL Analytics Endpoint
A T-SQL query interface over a Fabric Lakehouse. Read-only for DDL (cannot CREATE TABLE). dbt uses this to materialise views and run SELECT queries.
Delta / Parquet
Delta Lake is a storage layer using Parquet files + a transaction log. Parquet is a columnar file format (vs CSV rows). Much faster for analytical scans. Native format of MS Fabric Lakehouses.
Direct Lake
Power BI mode that reads directly from Delta files in OneLake — no import/refresh needed. Requires physical Delta tables. Not available for SQL views (use Import mode instead).
Import mode
Power BI loads data from the source into its in-memory VertiPaq engine. Fast for queries; requires periodic refresh to pick up new data. Used in this project (views → SQL Server connector).
VertiPaq
Power BI's in-memory columnar database engine. Stores data imported into the semantic model. Optimised for star schema queries and DAX measures.
DAX
Data Analysis Expressions. Power BI's formula language for calculated measures, columns, and table functions. Analogous to Excel formulas but for analytical data models.
Findur
Openlink Findur. Enterprise treasury and trading system. Used by banks and central banks for deal booking, position management, and risk reporting. The BIS source system.
BIS fiscal year
April 1 to March 31. Labelled FY2023-24, FY2024-25, etc. Different from calendar year. All BIS annual report figures use this convention.
CROSS JOIN
SQL operation producing the Cartesian product of two tables. Every row in table A is paired with every row in table B. Used in mart_maturity_ladder: 700 deals × 24 dates = 16,800 rows.
TRY_CAST
T-SQL function. Converts a value to a target data type. Returns NULL on failure (instead of throwing an error like CAST). Essential for working with raw string data from source systems.
EOMONTH(date, n)
T-SQL function returning the last day of the month, optionally offset by n months. EOMONTH('2023-04-15', -1) = '2023-03-31'. Used for FX rate date matching and as-of date generation.
date_key
Integer representation of a date: YEAR × 10000 + MONTH × 100 + DAY. Example: 2023-04-03 → 20230403. Avoids FORMAT() function (unavailable on Fabric endpoint). Sortable as integer.
ref() / source()
dbt Jinja macros. ref('model_name') declares a dependency on another dbt model. source('schema', 'table') declares a dependency on an external source table. These build the DAG.
ISO 3166-1 alpha-2
Two-letter country codes: CN (China), JP (Japan), US (United States), etc. Used in cpty_country and the region CASE statement in dim_counterparty.
ISO 4217
Three-letter currency codes: USD, EUR, GBP, JPY, CNY, CHF, etc. Used in the currency column of src_deals and src_fx_rates.
as-of date
The reference date for a snapshot calculation. "As of 31 March 2025" means: given the state of the world on that date. In the maturity ladder, it is the date from which days_to_maturity is calculated.
VALIDATED / MATURED / CANCELLED
BIS deal lifecycle states. VALIDATED = active, live deal. MATURED = reached maturity date and was repaid. CANCELLED = cancelled before maturity. Analytics marts filter to VALIDATED only.
